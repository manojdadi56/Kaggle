VERIFIED via the author's public GitHub repo (github.com/tonghuikang/nemotron, default branch master) — the Kaggle discussion body itself is login-gated (WebFetch returned only the page title), but the repo README, training code, and configs reproduce the full method. Tong Hui Kang (huikang) won the Open Progress Prize of the NVIDIA Nemotron Model Reasoning Challenge.

Core approach: rather than RL, treat the benchmark as a set of 9 puzzle categories (bit_manipulation, cipher, two cryptarithm variants, two equation-numeric variants, gravity, numeral, unit_conversion). For each category he wrote a deterministic Python solver that discovers the underlying rule, then emits a clean chain-of-thought trace mirroring that solver logic. These traces (plus auxiliary augmentation tasks: spelling, concatenation, splitting, matching, lstrip) become a synthetic SFT corpus formatted as '(reasoning)</think>\\boxed{answer}<|im_end|>'. He then LoRA-fine-tunes nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-BF16 (a 30B MoE with ~3B active params) to reproduce the reasoning.

Verbatim config (train_sft.py): LoRA rank 32, lr 2e-4 with step-linear decay, batch 64 (micro 16), 1 epoch, max_length 8192, Adam(0.9/0.95), train MLP+attn+unembed, cross_entropy loss. The repo also ships experimental RL losses (importance_sampling, PPO, CISPO, DRO) and a branch/cutoff-weighted CE, but the default winning run is plain SFT. Pipeline order: reasoning.py -> augmentation.py -> corpus.py -> train_sft.py -> modal upload_adapter.py.

The data/corpus-generation half (pure Python solvers) runs on any CPU, but the SFT step requires external managed compute or a much smaller model swap.

### tonghuikang/nemotron — Progress Prize winning submission (GitHub)
- **type:** external  |  **content:** fetched

- **score:** Kaggle LB 0.85 (notebook titled "end-to-end-finetuning-for-lb-0-85"); won a Progress Prize.
- **reproducible:** Not on free Kaggle 2xT4. Base model is nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-BF16 (30B-total MoE hybrid Mamba-Transformer, ~3.5B active) in BF16 — far exceeds 2x16GB T4 for LoRA SFT at seq_len 8192, batch 64. Training is offloaded to managed/serverless GPU backends (Tinker or Modal), not local Kaggle GPUs. HOWEVER, the *winning idea* is highly reproducible cheaply: the synthetic-data generation (reasoning.py / augmentation.py) is pure deterministic Python with no GPU and no LLM calls — only the final LoRA fine-tune needs a big-GPU backend. A smaller Nemotron variant could plausibly be LoRA-tuned on the same corpus on T4-class hardware as an experiment.
- **techniques:** Reverse-engineer the rules: competition tasks are programmatically-generated reasoning problems grouped by category (gravity, unit_conversion, cipher, string ops, etc.). Approach figures out each generator's underlying rule rather than relying on the LLM to reason it out.; Deterministic synthetic solvers (reasoning.py): per-category Python solvers in a GENERATORS dict produce ground-truth reasoning traces; answers extracted via \\boxed{...} regex and verified against stored answers (1e-2 rel tolerance for numbers, case-insensitive for strings). Problems tagged rule_found / rule_unknown / hypothesis_formed.; Templated data augmentation (augmentation.py): five rule-based augmenter modules (spelling, concatenation, splitting, matching, lstrip) emit prompt/completion pairs — fully deterministic, no LLM/API calls.; Corpus assembly (corpus.py) into prompt/completion training entries with masked/unmasked/total token accounting.; Supervised fine-tuning only — no RL in the actual recipe (train_sft.py exposes PPO/CISPO/DRO/importance-sampling loss options but the recipe uses cross-entropy SFT).; LoRA config: rank 32, target = MLP + attention + unembedding (train_mlp/train_attn/train_unembed=True); base model nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-BF16.; Training hyperparams: LR 2e-4, StepLinearDecayLRSchedule, batch 64 (micro-batch 16 on Modal), 1 epoch, max_seq_len 8192, Adam (b1=0.9, b2=0.95, eps=1e-8, wd=0.0, grad_clip 1e9), BF16.; Training backends: Tinker (managed LoRA service) or Modal (serverless GPU); adapter pushed via upload_adapter.py.; Evaluation/observability: interactive dashboards (Base/Synthetic/Corpus/Training/Metrics at nemotron.huikang.dev) showing per-problem solve status pre/post fine-tune, token-level logprob traces, per-epoch per-problem metrics, and run-level loss charts.

Progress-Prize-winning GitHub writeup for the Kaggle "NVIDIA Nemotron Model Reasoning Challenge" (Kaggle LB ~0.85). WebFetch on the GitHub repo and raw source files succeeded (the linked Kaggle notebook itself is JS-gated and yielded only its title + the 0.85 score). Core insight: the benchmark's tasks are programmatically generated per-category reasoning problems, so instead of trying to make the model reason better, the author reverse-engineers each category's generation rule and writes deterministic Python solvers (reasoning.py, GENERATORS dict over categories like gravity/unit_conversion/cipher/string-ops) plus five rule-based augmenter modules (augmentation.py) to mass-produce correct reasoning traces — with NO LLM calls in the data pipeline. That synthetic corpus is then used for a single-epoch LoRA SFT (corpus.py -> train_sft.py -> upload_adapter.py). Concrete training config: base model nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-BF16; LoRA rank 32 over MLP+attention+unembedding; LR 2e-4 with StepLinearDecayLRSchedule; batch 64 (micro 16 on Modal); 1 epoch; max_seq_len 8192; Adam (0.9/0.95, eps 1e-8, wd 0); BF16. It is SFT-only (no RL despite PPO/CISPO/DRO loss hooks being present). Training runs on managed/serverless GPU backends (Tinker or Modal) — a 30B BF16 model does NOT fit free Kaggle 2xT4, so the fine-tune is not T4-reproducible as-is, though the cheap, GPU-free synthetic-data idea is the transferable lesson and could be paired with a smaller model for T4 experiments.
